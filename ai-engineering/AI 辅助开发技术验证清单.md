# AI 辅助开发技术验证清单

## 一、开发前：先由我定义问题

### 1. 业务目标

* [ ] 这个功能解决什么真实问题？
* [ ] 谁会使用这个功能？
* [ ] 用户操作完成后，系统应该达到什么结果？
* [ ] 什么情况算成功？
* [ ] 什么情况绝对不能发生？
* [ ] 如果不做这个功能，会造成什么损失？

### 2. 范围边界

* [ ] 本次必须完成什么？
* [ ] 哪些内容明确不在本次范围内？
* [ ] 是否存在可以复用的现有能力？
* [ ] 是否真的需要新增表、状态、接口、定时任务或中间件？
* [ ] 有没有更简单的最小闭环方案？

### 3. 验收标准

在让 AI 编码前，我必须先写清楚：

* [ ] 正常场景
* [ ] 异常场景
* [ ] 边界场景
* [ ] 权限要求
* [ ] 幂等要求
* [ ] 并发要求
* [ ] 历史数据兼容要求
* [ ] 前后端展示要求

---

## 二、分析阶段：先读懂现状，再允许修改

要求 AI 先只读扫描，不立即改代码，并输出：

* [ ] 当前调用链
* [ ] 涉及的 Controller、Service、Mapper、SQL
* [ ] 涉及的数据表和核心字段
* [ ] 当前状态流转
* [ ] 事务边界
* [ ] 权限校验位置
* [ ] 幂等机制
* [ ] 已有异常处理
* [ ] 当前实现存在的问题
* [ ] 最小修改方案
* [ ] 预计修改文件清单

我需要确认：

* [ ] AI 对需求的理解是否正确
* [ ] AI 对现有代码的理解是否正确
* [ ] 是否遗漏现有能力
* [ ] 是否出现重复建设
* [ ] 是否把简单问题复杂化
* [ ] 是否存在未经证实的推测

任何关键结论都必须要求 AI 提供：

* 文件路径
* 类名
* 方法名
* 字段名
* SQL 条件
* 配置位置

不能只接受“已经支持”“已经保证”“不会发生”这类结论。

---

## 三、设计阶段：由我掌握方案选择权

### 1. 数据与状态

* [ ] 事实来源是什么？
* [ ] 哪张表或哪个系统的状态最可信？
* [ ] 谁拥有状态修改权？
* [ ] 合法状态有哪些？
* [ ] 允许哪些状态迁移？
* [ ] 是否可能出现中间状态？
* [ ] 中间状态如何恢复？

### 2. 事务

* [ ] 事务从哪个方法开始？
* [ ] 哪些数据库操作在同一个事务中？
* [ ] 是否存在跨数据源操作？
* [ ] 是否存在异步、事件、消息队列或远程调用？
* [ ] 异常是否会正常抛出并触发回滚？
* [ ] 是否存在事务失效的内部方法调用？

### 3. 幂等

* [ ] 幂等键是什么？
* [ ] 是否存在数据库唯一约束？
* [ ] 是否使用条件更新？
* [ ] 重复请求会发生什么？
* [ ] 重复回调会发生什么？
* [ ] 定时任务重复执行会发生什么？
* [ ] 多实例并发执行会发生什么？
* [ ] 分布式锁失效后，数据库是否仍然安全？

### 4. 权限

* [ ] 当前用户身份从哪里获取？
* [ ] 资源归属通过什么字段判断？
* [ ] 权限校验是在 Controller 还是 Service？
* [ ] 列表、详情、下载、修改、删除是否都校验？
* [ ] 是否能够通过修改 ID 访问他人数据？
* [ ] 管理员权限和普通用户资源权限是否混淆？

### 5. 复杂度

每新增一个机制前必须回答：

* [ ] 不新增会发生什么？
* [ ] 发生概率多大？
* [ ] 最坏损失多大？
* [ ] 当前机制是否已经覆盖？
* [ ] 新机制会引入什么维护成本？
* [ ] 是否可以先做更小的方案？

---

## 四、编码阶段：AI 执行，我控制变更

要求 AI 在修改前明确：

* [ ] 修改哪些文件
* [ ] 每个文件为什么需要修改
* [ ] 哪些文件明确不修改
* [ ] 是否修改接口
* [ ] 是否修改数据库
* [ ] 是否影响历史数据
* [ ] 是否影响其他模块
* [ ] 是否需要新增配置
* [ ] 是否具备回滚方案

编码过程中坚持：

* [ ] 优先最小改动
* [ ] 不做无关重构
* [ ] 不擅自升级依赖
* [ ] 不修改无关配置
* [ ] 不新增未经确认的业务逻辑
* [ ] 不复制已有公共能力
* [ ] 核心业务逻辑不能只写在 Controller 或 Handler
* [ ] 状态值不能使用难以理解的魔法数字
* [ ] 异常不能被无意义吞掉

---

## 五、代码审查：不能只看“能不能运行”

我至少需要亲自检查：

* [ ] Git diff 是否只有预期变更
* [ ] 是否存在无关文件
* [ ] 是否误改配置
* [ ] 是否误删已有逻辑
* [ ] 核心查询条件是否正确
* [ ] 更新语句的 WHERE 条件是否安全
* [ ] 是否可能全表更新或全表删除
* [ ] 是否可能重复插入
* [ ] 是否存在空值问题
* [ ] 是否存在异常被吞
* [ ] 是否存在事务失效
* [ ] 是否存在越权入口
* [ ] 是否存在状态被任意覆盖
* [ ] 是否存在任务职责重叠
* [ ] 是否出现明显过度设计

当 AI 声称以下结论时，必须继续验证：

| AI 结论   | 必须检查                   |
| ------- | ---------------------- |
| 已保证幂等   | 唯一索引、条件更新、事务、并发结果      |
| 已保证事务   | 事务入口、代理调用、数据源、异常传播     |
| 已防止越权   | 所有入口、Service 校验、资源归属字段 |
| 已支持重试   | 重试范围、次数、退避、重复执行安全性     |
| 已保证并发安全 | 锁范围、锁失效、数据库兜底          |
| 已完成测试   | 测试覆盖场景和真实断言            |
| 已兼容历史数据 | 旧数据样本、默认值、迁移方式         |

## 六、测试阶段：测试标准由我定义

### 必测场景

* [ ] 正常执行一次
* [ ] 同一请求重复执行
* [ ] 两个请求并发执行
* [ ] 中间步骤异常
* [ ] 数据不存在
* [ ] 状态不合法
* [ ] 参数为空
* [ ] 参数越界
* [ ] 无权限访问
* [ ] 历史数据访问
* [ ] 服务重启后恢复
* [ ] 定时任务重复触发
* [ ] 第三方接口超时
* [ ] 数据库操作失败

### 验证证据

不能只看前端提示，需要检查：

* [ ] 接口响应
* [ ] 数据库最终状态
* [ ] 是否产生重复记录
* [ ] 是否出现错误状态
* [ ] 日志是否能关联业务 ID
* [ ] 事务是否按预期回滚
* [ ] 文件或对象存储是否一致
* [ ] 重复执行结果是否一致
* [ ] 权限绕过测试是否失败
* [ ] 异常后是否可以安全恢复

## 七、完成后：把 AI 产出转化成我的能力

每个重要功能完成后，我必须能够脱离 AI 回答：

1. 这个功能解决什么业务问题？
2. 正常调用链怎么走？
3. 涉及哪些核心表和字段？
4. 状态如何流转？
5. 事务边界在哪里？
6. 如何保证幂等？
7. 如何处理并发？
8. 权限边界在哪里？
9. 最容易失败的环节是什么？
10. 失败后如何恢复？
11. 为什么选择当前方案？
12. 哪些方案被放弃，为什么？

如果这些问题讲不清楚，说明我只是让 AI 完成了功能，还没有真正掌握它。

## 九、自我掌握度评估

### L1：AI 操作级

* 会描述需求
* 会让 AI 修改
* 会运行项目
* 会查看页面效果
* 出错后主要依赖 AI 排查

### L2：工程理解级

* 能解释正常调用链
* 能阅读关键代码
* 能检查 SQL 和状态
* 能自己定义测试场景
* 能发现明显设计问题
* 能通过日志和数据库定位问题

### L3：技术负责级

* 能定义状态机和职责边界
* 能解释事务和并发问题
* 能设计幂等与故障恢复
* 能判断方案是否过度设计
* 能决定哪些机制不应该增加
* 能在事故中快速定位并安全修复
* 能独立向别人讲清楚设计取舍

目标不是减少使用 AI，而是逐渐把重要模块从 L1、L2 提升到 L3。

---

## 十、三条强制规则

1. **没有先理解业务和现状，不允许 AI 直接改代码。**
2. **没有代码、SQL、日志或测试证据，不接受 AI 的结论。**
3. **功能完成后讲不清业务、状态、事务、幂等和失败恢复，就不算真正完成。**

最终工作模式：

> 我定义目标和边界，AI 提供分析与执行，我负责判断、验证和验收。
