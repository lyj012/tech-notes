# 后端工程化每日训练 Day 05：延迟队列、ACK、重试与死信队列复盘

## 一、今日主题

今天学习的核心知识点是：

> 延迟队列，以及它与 ACK、有限重试、死信队列、幂等机制之间的配合关系。

延迟队列主要解决：

> 某个业务不需要立即执行，而是需要在未来某个时间点触发处理。

常见业务场景包括：

* 订单超过 30 分钟未支付，自动取消
* AI 异步任务运行超过 20 分钟，自动标记超时
* PDF 临时文件保留 24 小时后删除
* 会员到期后失效
* 优惠券和验证码过期

---

## 二、为什么需要延迟队列

以 AI 异步任务超时为例。

任务状态流转：

```text
CREATED
→ RUNNING
→ SUCCESS / FAILED / TIMEOUT
```

系统要求：

```text
任务进入 RUNNING 后
20 分钟仍未完成
自动标记为 TIMEOUT
```

一种简单实现是每分钟扫描数据库：

```sql
SELECT *
FROM ai_task
WHERE status = 'RUNNING'
  AND start_time < NOW() - INTERVAL 20 MINUTE;
```

当任务数量很少时，定时扫描可以使用。

但当任务量达到几十万、上百万时，每分钟扫描大量数据会产生：

* 大量无效查询
* 数据库索引压力
* 磁盘和 CPU 消耗
* 定时任务执行时间越来越长
* 多实例执行时的重复处理问题

实际上，每次可能只有少量任务真正超时，却需要检查大量没有超时的任务。

延迟队列的思路是：

```text
每个任务进入 RUNNING
→ 发送一条 20 分钟延迟消息
→ 20 分钟后只检查这个 taskId
```

它将：

```text
定时全量扫描
```

转换为：

```text
按任务定点检查
```

---

## 三、AI 任务超时的完整流程

### 1. 任务开始执行

任务状态更新为：

```text
RUNNING
```

然后发送一条延迟消息：

```json
{
  "taskId": 1001
}
```

消息延迟时间：

```text
20 分钟
```

---

### 2. 消息进入延迟队列

消息不会立即被消费者处理，而是在延迟队列中等待。

```text
Producer
→ Delay Queue
→ 等待 20 分钟
```

20 分钟到期后，消息进入真正的执行队列：

```text
Delay Queue
→ Execute Queue
→ Consumer
```

延迟队列只负责：

> 控制消息什么时候开始被处理。

它不能保证消费者的业务逻辑一定执行成功。

---

### 3. 消费者查询任务状态

消费者收到消息后，第一步不能直接把任务改为 `TIMEOUT`，而是必须重新查询数据库中的当前状态。

可能出现三种情况。

#### 情况一：任务仍然是 RUNNING

说明任务可能已经超时。

消费者执行条件更新：

```sql
UPDATE ai_task
SET status = 'TIMEOUT',
    finish_time = NOW()
WHERE id = 1001
  AND status = 'RUNNING';
```

更新成功后：

```text
提交数据库事务
→ 发送 ACK
```

---

#### 情况二：任务已经 SUCCESS

例如任务在第 18 分钟已经执行成功。

第 20 分钟延迟消息到达后，消费者查询发现：

```text
status = SUCCESS
```

此时：

```text
不修改任务状态
→ 直接 ACK
```

这里 ACK 的含义不是“任务超时处理成功”，而是：

> 这条“检查任务是否超时”的消息已经处理完成。

检查后发现无需处理，也属于消费成功。

---

#### 情况三：任务已经 FAILED 或 TIMEOUT

同样不需要再次修改：

```text
查询当前状态
→ 判断无需处理
→ ACK
```

---

## 四、为什么要使用条件更新

错误写法：

```sql
UPDATE ai_task
SET status = 'TIMEOUT'
WHERE id = 1001;
```

这种写法存在并发覆盖问题。

例如：

```text
消费者查询到 RUNNING
→ Worker 同时将任务更新为 SUCCESS
→ 消费者又无条件更新为 TIMEOUT
```

最终会把成功任务错误覆盖成超时任务。

正确写法：

```sql
UPDATE ai_task
SET status = 'TIMEOUT'
WHERE id = 1001
  AND status = 'RUNNING';
```

这条 SQL 只允许：

```text
RUNNING → TIMEOUT
```

如果任务已经变成：

```text
SUCCESS
FAILED
TIMEOUT
```

则影响行数为 0，不会覆盖已有状态。

这既是状态机控制，也是当前场景下最直接的幂等实现。

---

## 五、ACK 的作用

RabbitMQ 不知道消费者的数据库操作是否执行成功。

消费者处理完成后，需要告诉 RabbitMQ：

```text
这条消息已经处理完成，可以删除
```

这就是 ACK。

正常顺序应该是：

```text
收到消息
→ 执行业务逻辑
→ 数据库事务提交成功
→ ACK
→ MQ 删除消息
```

不能先 ACK，再执行数据库操作。

错误流程：

```text
收到消息
→ 先 ACK
→ MQ 删除消息
→ 数据库操作失败
```

结果是：

```text
消息已经丢失
但任务仍然没有更新为 TIMEOUT
```

因此：

> 业务处理成功并且事务提交后，才能确认当前消息已完成。

---

## 六、业务处理失败时怎么办

假设延迟消息到达后，消费者准备更新任务状态，但数据库宕机。

流程：

```text
收到消息
→ 查询或更新数据库
→ 数据库连接失败
```

此时不能直接认为业务处理成功。

系统需要进入重试流程。

---

## 七、为什么不能无限立即重试

一种简单做法是：

```text
NACK
→ requeue = true
→ 消息重新进入原队列
```

但数据库仍然宕机时，可能形成：

```text
失败
→ 立即重新消费
→ 再次失败
→ 立即重新消费
```

这种无限重试会导致：

* CPU 空转
* 错误日志快速增长
* 数据库持续受到连接请求冲击
* 异常消息长期占用消费者
* 正常消息无法及时处理
* 消息队列积压

因此，重试必须满足两个条件：

```text
有限次数
+
延迟重试
```

---

## 八、有限重试设计

可以设计为：

```text
第一次失败
→ 10 秒后重试

第二次失败
→ 1 分钟后重试

第三次失败
→ 5 分钟后重试

超过最大次数
→ 进入死信队列
```

消息中可以携带重试次数：

```json
{
  "taskId": 1001,
  "retryCount": 2
}
```

消费者根据次数进行判断：

```text
retryCount < maxRetryCount
→ 投递到重试队列

retryCount >= maxRetryCount
→ 投递到死信队列
```

重试间隔可以逐渐增加，避免系统故障期间持续冲击数据库。

这种方式属于退避重试。

---

## 九、业务失败后为什么有时仍然 ACK

这是今天最容易混淆的地方。

业务处理失败时，可以将当前消息重新发送到延迟重试队列。

流程：

```text
当前消息消费失败
→ 生成新的重试消息
→ 发送到重试队列
→ 确认重试消息发送成功
→ ACK 当前消息
```

虽然业务没有最终成功，但当前消息的责任已经转移到了重试消息。

此时 ACK 的含义是：

> 当前这一次消息投递已经被妥善处理，后续由重试队列继续负责。

如果不 ACK 原消息，可能同时出现：

```text
原消息被 MQ 再次投递
+
重试队列中的消息再次投递
```

从而产生重复消费。

因此，正确顺序是：

```text
先确保重试消息发送成功
→ 再 ACK 原消息
```

错误顺序：

```text
先 ACK 原消息
→ 再发送重试消息
→ 重试消息发送失败
```

这样会导致消息彻底丢失。

在要求较高的系统中，还需要结合 Publisher Confirm，确认重试消息确实已经发送到 RabbitMQ。

---

## 十、死信队列的作用

当消息超过最大重试次数后，不应该继续无限重试。

此时将消息放入死信队列：

```text
task.timeout.dlq
```

死信队列的作用不是继续疯狂重试，而是：

> 隔离持续失败的异常消息，防止它们影响正常消息处理。

进入死信队列的原因可能包括：

* 数据库长时间不可用
* 消息数据错误
* taskId 不存在
* 业务代码存在 Bug
* 状态机逻辑异常
* 数据库字段或约束异常

死信消息可以包含：

```json
{
  "taskId": 1001,
  "retryCount": 5,
  "failureReason": "Database connection timeout",
  "failedAt": "2026-07-12T10:30:00"
}
```

后续处理方式包括：

* 触发监控告警
* 在后台管理系统展示
* 定时补偿程序重新处理
* 人工确认后重新投递
* 修复程序后批量重放

死信队列不是垃圾桶，而是：

```text
异常消息隔离区
```

---

## 十一、死信队列中的消息如何恢复

消息进入死信队列后，原消费者不会一直等待人工处理。

原消费者已经结束本次消费。

后续可以由补偿程序或人工操作完成：

```text
读取 DLQ 消息
→ 检查故障是否恢复
→ 重新投递到执行队列
→ 再次执行统一消费逻辑
```

相比人工直接修改数据库，更推荐重新投递消息。

因为重新走消费逻辑可以继续复用：

* 状态检查
* 条件更新
* 幂等控制
* 数据库事务
* 日志记录
* 异常处理

---

## 十二、为什么还需要幂等

MQ 通常不能保证一条消息永远只投递一次。

例如：

```text
消费者已经更新数据库成功
→ 准备发送 ACK
→ 网络断开
→ RabbitMQ 没有收到 ACK
→ 消息再次投递
```

这时数据库中的任务已经是：

```text
TIMEOUT
```

但同一条消息又被消费者收到。

消费者必须保证重复消费不会产生错误副作用。

当前任务超时场景中，可以通过条件更新实现幂等：

```sql
UPDATE ai_task
SET status = 'TIMEOUT'
WHERE id = ?
  AND status = 'RUNNING';
```

第一次执行：

```text
RUNNING → TIMEOUT
影响 1 行
```

第二次重复执行：

```text
当前已经是 TIMEOUT
不满足 status = RUNNING
影响 0 行
```

消费者发现无需再次处理后，直接 ACK。

因此：

```text
MQ 负责消息投递和重新投递
消费者负责幂等处理
```

幂等逻辑应该设计在消费者和业务数据库中，而不是“加在 MQ 上”。

---

## 十三、两种容易混淆的死信

RabbitMQ 使用 `TTL + Dead Letter Exchange` 实现延迟队列时，也会使用死信机制。

但这和业务失败后的死信队列不是同一个概念。

### 1. 用于实现延迟的死信转发

```text
消息进入延迟队列
→ TTL 到期
→ 通过死信交换机转发
→ 进入执行队列
```

这里消息不是因为业务执行失败。

只是利用 RabbitMQ 的死信转发机制实现：

```text
等待指定时间后再消费
```

---

### 2. 业务多次失败后的死信队列

```text
消费者处理失败
→ 多次有限重试
→ 仍然失败
→ 进入最终 DLQ
```

这里才是异常消息隔离。

因此，系统中可能同时存在：

```text
task.timeout.delay.queue
task.timeout.execute.queue
task.timeout.retry.queue
task.timeout.dlq
```

它们分别承担不同责任。

---

## 十四、完整可靠性链路

AI 任务超时处理的完整流程可以整理为：

```text
任务进入 RUNNING
→ 发送 20 分钟延迟消息
→ 消息等待 20 分钟
→ 消费者收到消息
→ 查询任务当前状态
```

如果状态已经是：

```text
SUCCESS / FAILED / TIMEOUT
```

则：

```text
不修改
→ ACK
```

如果仍然是：

```text
RUNNING
```

则执行：

```text
WHERE status = 'RUNNING'
条件更新为 TIMEOUT
```

更新成功：

```text
数据库事务提交
→ ACK
```

更新失败或数据库异常：

```text
发送到延迟重试队列
→ 确认重试消息发送成功
→ ACK 当前消息
```

超过最大重试次数：

```text
发送到死信队列
→ 告警
→ 补偿或人工重新投递
```

---

## 十五、各机制分别解决什么问题

```text
延迟队列
→ 解决什么时候处理

状态检查
→ 解决业务状态可能已经变化

条件更新
→ 解决并发覆盖和状态机约束

幂等
→ 解决重复消费和重复副作用

ACK
→ 让 MQ 知道当前消息是否已经妥善处理

重试
→ 解决数据库抖动等临时故障

死信队列
→ 隔离持续失败的异常消息

补偿机制
→ 处理自动重试仍无法解决的问题
```

## 十六、今天纠正的错误

### 错误一：认为幂等加在 MQ 上

错误理解：

```text
为了防止 ACK 丢失导致重复消费
在 MQ 上加幂等
```

正确理解：

```text
MQ 可能重复投递消息
消费者必须保证重复消费不会重复执行业务
```

幂等应设计在：

* 消费者业务逻辑
* 数据库唯一约束
* 状态机条件更新
* 消费记录表
* Redis 去重记录

当前 AI 任务场景中，使用：

```sql
WHERE status = 'RUNNING'
```

即可实现基础幂等。

---

### 错误二：认为业务失败就绝对不能 ACK

更准确的判断是：

```text
业务失败，并且消息没有安全转移
→ 不能 ACK

业务失败，但已经成功转移到重试队列或 DLQ
→ 可以 ACK 当前消息
```

ACK 表示的是：

> 当前消息已经被妥善处理，而不一定表示最终业务已经成功。

---

### 错误三：认为进入死信队列后消费者继续等待人工处理

正确流程是：

```text
消息进入 DLQ
→ 当前消费者结束
→ 后续由补偿任务、后台系统或人工重新投递
```

消费者不能一直占用线程，等待人工几个小时或几天。

## 十七、面试表达

在需要未来某个时间点执行的业务中，我会考虑使用延迟队列，例如订单超时、AI 任务超时、会员到期和临时文件清理。

以 AI 异步任务为例，任务进入 `RUNNING` 状态后发送一条 20 分钟延迟消息。消息到期后，消费者不会直接把任务更新成 `TIMEOUT`，而是先查询任务当前状态。如果任务已经成功或失败，则直接 ACK；如果仍然处于 `RUNNING`，则使用带状态条件的 SQL 将其更新为 `TIMEOUT`，避免并发情况下覆盖已经成功的任务。

如果数据库操作因为临时故障失败，我会采用有限次数的延迟重试，而不是立即无限重新入队。确认重试消息发送成功后，再 ACK 当前消息。超过最大重试次数后，将消息放入死信队列，通过监控告警、定时补偿或人工重新投递进行处理。

同时，消费者必须保证幂等，因为数据库操作成功但 ACK 丢失时，MQ 可能再次投递相同消息。可以通过状态机条件更新、唯一键或消费记录表防止重复处理，从而实现最终一致性。

---

## 十八、今日最终理解

延迟队列不能单独保证业务可靠执行。

完整设计通常需要结合：

```text
延迟队列
+
状态检查
+
条件更新
+
幂等
+
ACK
+
有限重试
+
死信队列
+
补偿机制
```

最终需要记住：

> 延迟队列保证晚一点处理，ACK 告诉 MQ 当前消息是否已被妥善处理，重试解决临时故障，死信队列隔离持续失败的消息，幂等防止重复消费产生重复副作用。

以及：

> MQ 负责消息投递和重新投递，消费者负责业务幂等。
