# 后端工程化每日训练 Day 07：事务边界与数据库、MQ 最终一致性复盘

## 一、今日主题

今天学习的核心问题是：

> 为什么数据库事务不能解决整个业务流程中的所有一致性问题？

在实际后端系统中，一个完整业务流程通常不只涉及数据库，还可能包含：

* 消息队列；
* Redis；
* 文件存储；
* 第三方支付；
* 短信和邮件；
* AI 模型接口；
* 其他微服务。

Spring 的 `@Transactional` 主要用于管理数据库本地事务，它无法直接回滚已经发送的 MQ 消息，也无法撤销已经发生的远程接口调用。

因此，真实系统中的一致性问题，通常不能只依靠数据库事务解决。

---

## 二、事务真正解决什么问题

数据库事务主要保证同一个事务资源中的一组数据库操作，要么全部成功，要么全部失败。

例如转账操作：

```text
扣减用户 A 的余额
↓
增加用户 B 的余额
```

如果两个 SQL 属于同一个事务，那么应该满足：

```text
全部提交
```

或者：

```text
全部回滚
```

不能只执行其中一半。

但是，事务无法天然保证下面这些外部操作的一致性：

```text
数据库写入
+
发送 RabbitMQ
+
调用 GPT
+
上传 OSS
+
发送短信
```

所以更准确的理解是：

> 本地事务主要保证数据库内部的一致性，跨系统业务流程通常需要通过状态机、消息队列、幂等、重试和补偿机制实现最终一致性。

---

## 三、为什么不能把整个 AI 任务流程放进一个事务

AI 异步任务平台的基本流程如下：

```text
创建任务
↓
保存 Task
↓
发送 MQ
↓
Worker 调用 GPT
↓
保存执行结果
```

不能直接写成：

```java
@Transactional
public void executeTask() {
    saveTask();
    sendMq();
    callGpt();
    saveResult();
}
```

主要有以下几个原因。

### 1. `@Transactional` 不能控制 GPT 和 MQ

即使 GPT 调用失败，数据库事务也无法撤销 GPT 已经发生的调用。

同样，即使数据库最终回滚，已经发送到 RabbitMQ 的消息也不会自动消失。

---

### 2. 远程接口耗时不可控

GPT 调用可能耗时几十秒，甚至发生超时。

如果把 GPT 调用放进事务中，会导致：

* 数据库连接长时间被占用；
* 已修改的数据可能持续持有锁；
* 事务超时概率增加；
* 并发请求可能被阻塞；
* 数据库连接池可能耗尽。

因此事务应该尽量短，只包含必须保证原子性的数据库操作。

---

### 3. 外部操作无法被数据库回滚

例如：

```text
GPT 调用成功
↓
数据库保存结果失败
```

即使数据库事务回滚，GPT 调用产生的费用和结果也已经真实发生，数据库无法撤销这次调用。

---

## 四、今日思考题与我的理解

## 1. 为什么不能把整个流程放进一个 `@Transactional` 方法

我的初步理解是：

`@Transactional` 主要保证数据库操作的一致性，无法保证消息队列、第三方接口和远程服务的一致性。

例如：

```text
数据库保存任务成功
↓
调用 GPT 超时
```

数据库中已经存在任务，但 GPT 没有产生结果，数据库事务不能自动解决这种跨系统问题。

进一步补充后，我认识到还有一个重要原因：

> GPT 调用属于耗时远程操作，把它放入事务中会导致事务范围过大，长时间占用数据库连接和锁资源。

因此，更合理的拆分方式是：

```text
请求线程：
使用短事务创建任务

异步 Worker：
调用 GPT

保存结果：
再使用一个短事务
```

---

## 2. 数据库已经提交，但 MQ 发送失败

流程如下：

```text
数据库保存 Task 成功
↓
数据库事务提交
↓
发送 MQ 失败
```

此时数据库中已经存在任务，例如：

```text
task_id = 1001
status = CREATED
```

但是 MQ 中没有对应消息，消费者根本不会收到任务。

最终结果是：

```text
任务一直停留在 CREATED
↓
没有 Worker 执行
↓
不会调用 GPT
↓
不会产生最终结果
```

这说明：

> 消费者幂等无法解决生产者没有成功发送消息的问题，因为消费者根本没有收到消息。

---

## 3. MQ 已经发送成功，但数据库最终回滚

流程如下：

```text
保存 Task
↓
发送 MQ 成功
↓
数据库事务提交失败并回滚
```

最终可能出现：

```text
MQ 中存在 taskId = 1001
数据库中不存在 taskId = 1001
```

消费者收到消息后，可能发生两种情况。

### 情况一：消费者先查询任务

消费者发现任务不存在：

```text
Task not found
```

消息可能不断重试，最终进入死信队列。

### 情况二：消费者没有校验，直接调用 GPT

可能出现：

```text
GPT 已经被调用
↓
产生调用费用
↓
数据库中却没有对应任务
↓
结果无法正常关联和保存
```

这会产生：

* 无效模型调用；
* 额外成本；
* 重复重试；
* 孤立执行结果；
* 死信消息；
* 排查困难。

---

## 五、我最初的解决思路

我最初想到的方案包括：

```text
@Transactional
+
消费者幂等
+
补偿机制
```

这些方向是正确的，但还没有组成一套完整链路。

另外需要纠正一个表达：

我之前语音中说的 “MID”，实际指的是：

> 幂等。

幂等主要解决的是：

```text
同一条消息被重复消费
↓
业务不能重复执行
```

但幂等不能解决：

```text
数据库已经成功
但 MQ 消息根本没有发送出去
```

这是生产端可靠发送问题，需要另外设计。

---

## 六、数据库与 MQ 最终一致性的基本方案

一种常见方案是：

> 本地消息表，也叫 Transactional Outbox。

核心思想是：

```text
不直接要求数据库提交和 MQ 发送同时成功
而是先可靠记录“待发送消息”
再通过重试保证消息最终发送成功
```

---

## 七、本地消息表完整流程

## 第一步：同一个事务中保存任务和待发送消息

在一个本地数据库事务中，同时写入：

```text
ai_task
+
outbox_message
```

代码示例：

```java
@Transactional
public Long createTask(CreateTaskRequest request) {
    AiTask task = new AiTask();
    task.setStatus(TaskStatus.CREATED);
    taskMapper.insert(task);

    OutboxMessage message = new OutboxMessage();
    message.setBusinessId(task.getId());
    message.setMessageType("AI_TASK_CREATED");
    message.setStatus("PENDING");
    outboxMessageMapper.insert(message);

    return task.getId();
}
```

这里事务能够保证：

```text
Task 和消息记录全部成功
```

或者：

```text
Task 和消息记录全部回滚
```

不会出现：

```text
Task 创建成功
但系统完全不知道还需要发送 MQ
```

---

## 第二步：独立发送器发送 MQ

事务提交以后，由定时任务或独立消息发送器查询：

```text
status = PENDING
```

的消息记录。

处理流程：

```text
查询待发送消息
↓
发送 RabbitMQ
↓
发送成功
↓
更新消息状态为 SENT
```

如果发送失败：

```text
保持 PENDING
↓
记录重试次数
↓
等待下次重新发送
```

这样即使系统刚提交数据库就宕机，本地消息记录仍然存在，系统恢复后还可以继续发送。

---

## 第三步：为什么消费者仍然需要幂等

假设出现以下流程：

```text
MQ 已经发送成功
↓
程序准备把消息状态更新为 SENT
↓
程序突然宕机
```

数据库中消息状态仍然是：

```text
PENDING
```

系统恢复后，会再次发送同一条 MQ 消息。

因此：

> MQ 消息可能被重复投递，消费者必须进行幂等校验。

可以根据以下字段判断是否已经处理：

```text
messageId
```

或者：

```text
taskId + eventType
```

也可以建立消费记录表：

```text
consumer_message_log
```

并建立唯一索引：

```sql
UNIQUE KEY uk_message_id (message_id)
```

如果同一条消息再次到达，唯一索引插入失败，说明消息已经处理过，不再重复调用 GPT。

---

## 第四步：Worker 执行任务

消费者收到任务消息后，可以按照以下流程执行：

```text
收到 MQ 消息
↓
进行幂等校验
↓
查询任务是否存在
↓
检查任务当前状态
↓
更新状态为 RUNNING
↓
提交事务
↓
调用 GPT
↓
开启新事务
↓
保存执行结果
↓
更新状态为 SUCCESS
↓
记录消费完成
↓
提交事务
↓
发送 ACK
```

如果 GPT 调用失败：

```text
记录失败原因
↓
更新为 RETRYING 或 FAILED
↓
按照策略重试
↓
超过最大重试次数
↓
进入死信队列
```

---

## 八、不同问题对应不同解决方案

| 问题              | 主要解决方案          |
| --------------- | --------------- |
| 数据库成功，MQ 发送失败   | 本地消息表、定时补偿、事务消息 |
| MQ 消息重复发送       | 消费者幂等           |
| ACK 丢失导致重复消费    | 消费者幂等           |
| 消费者临时执行失败       | 重试机制            |
| 多次重试仍失败         | 死信队列            |
| GPT 成功但保存结果失败   | 状态机、结果补偿、幂等保存   |
| 任务长期停留在 CREATED | 定时扫描和补偿发送       |
| 任务长期停留在 RUNNING | 超时检测、延迟队列、补偿机制  |

核心认识是：

> 生产端可靠发送、消费端幂等和业务失败补偿，是三个不同的问题，不能只用一个机制解决。

---

## 九、事务的正确使用原则

### 1. 事务范围要尽量小

事务中只保留必须一起成功或失败的数据库操作。

```java
@Transactional
public Long saveTask(Task task) {
    taskMapper.insert(task);
    taskLogMapper.insertCreateLog(task.getId());
    return task.getId();
}
```

不要在事务中加入：

```text
调用 GPT
上传文件
发送短信
发送邮件
长时间循环
等待 MQ 响应
```

---

### 2. 不要吞掉异常

错误示例：

```java
@Transactional
public void createTask() {
    try {
        taskMapper.insert(task);
    } catch (Exception e) {
        log.error("创建任务失败", e);
    }
}
```

异常被捕获后没有继续抛出，Spring 可能认为方法正常结束，从而提交事务。

更合理的处理方式是：

```java
@Transactional
public void createTask() {
    try {
        taskMapper.insert(task);
    } catch (Exception e) {
        log.error("创建任务失败", e);
        throw e;
    }
}
```

---

### 3. 不要认为加了事务就绝对安全

即使使用了 `@Transactional`，仍然需要考虑：

* 方法是否由 Spring 代理调用；
* 是否发生同类内部调用；
* 异常类型是否触发回滚；
* 异常是否被吞掉；
* 数据库表是否支持事务；
* 事务传播行为是否正确；
* 事务范围是否过大；
* 外部系统操作是否需要补偿。

---

## 十、面试表达

面试时可以这样回答：

> `@Transactional` 主要解决本地数据库事务的一致性，不能直接保证数据库与 MQ、Redis、文件存储或第三方接口之间的原子性。如果任务保存成功但 MQ 发送失败，会导致任务长期停留在待执行状态；如果 MQ 已发送但数据库回滚，消费者可能收到一条不存在对应业务数据的消息。
>
> 在实际设计中，我会缩小事务范围，只把需要保证原子性的数据库操作放入事务。对于数据库和 MQ 的一致性，可以使用本地消息表，将业务数据和待发送消息放在同一个本地事务中保存，再由独立发送器或定时任务可靠发送 MQ，并通过重试和补偿保证消息最终送达。
>
> 由于消息可能重复投递，消费者还需要进行幂等校验，并结合任务状态机、失败重试、死信队列和超时补偿保证整个异步任务流程最终一致。

---

## 十一、今日核心结论

今天最重要的认识不是记住事务的 ACID，而是理解事务的边界：

```text
本地事务
解决数据库内部原子性

消息队列
解决异步和解耦

幂等
解决重复执行

重试
解决临时失败

死信队列
隔离无法继续处理的消息

补偿机制
修复已经发生的不一致

状态机
描述并控制业务流程

本地消息表
解决数据库与 MQ 的最终一致性
```

最终可以总结为：

> 数据库事务无法覆盖完整的分布式业务流程。真实后端系统通常通过短事务、状态机、可靠消息、消费者幂等、失败重试和补偿机制共同实现最终一致性。
