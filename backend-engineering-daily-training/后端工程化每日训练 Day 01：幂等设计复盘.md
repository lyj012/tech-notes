# 后端工程化每日训练 Day 01：幂等设计复盘

## 1. 今天学习的核心知识点

今天学习的是后端工程化中的 **幂等设计**。

幂等的核心含义是：

> 同一个业务请求无论执行多少次，最终业务结果都应该保持一致。

它主要解决的是线上系统中由于重复请求、重复回调、重复消费、客户端重试、网络超时等原因导致的 **重复业务执行问题**。

典型场景包括：

* 支付平台重复回调，导致会员、积分、订单状态被重复处理。
* 用户重复点击按钮，导致重复创建任务。
* MQ 消息重复投递，导致业务被重复消费。
* AI 异步任务因为网络超时被重复提交，导致模型重复调用、成本增加。
* 文件转换平台重复创建转换任务，浪费 CPU、GPU、存储等资源。

我今天理解到：
**幂等不是简单地写一个 if 判断，而是要从业务唯一标识、数据库约束、Redis 防重、状态机、事务和异常补偿等多个层面共同设计。**

---

## 2. 幂等的核心：Business Key

今天最大的收获是理解了 **Business Key（业务唯一标识）**。

以前我容易把“唯一索引”和“唯一标识”混在一起。现在理解为：

* `UNIQUE` 是数据库层面的唯一约束。
* `Business Key` 是业务层面用来判断“这是不是同一个业务请求”的稳定标识。

也就是说，真正重要的不是简单写一个 `UNIQUE`，而是要先设计出合理的业务唯一键。

例如 AI 异步任务平台中，Business Key 可以由以下信息组合生成：

```text
userId + taskType + fileHash + promptHash + model + paramsHash
```

然后再整体生成一个 hash：

```text
businessKey = sha256(userId + taskType + fileHash + promptHash + model + paramsHash)
```

这样，如果客户端在 1 秒内连续发送 3 次完全相同的请求，生成出来的 `businessKey` 是一样的，系统就能识别这是同一个业务请求，而不是 3 个不同任务。

错误设计是使用：

```java
UUID.randomUUID()
```

因为 UUID 每次都不一样，无法判断重复请求，等于没有幂等能力。

## 3. 数据库唯一索引的作用

数据库唯一索引适合作为幂等设计的最终兜底。

例如 AI 任务表可以设计：

```sql
CREATE TABLE ai_task (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    business_key VARCHAR(128) NOT NULL,
    task_type VARCHAR(64) NOT NULL,
    status VARCHAR(32) NOT NULL,
    result TEXT,
    error_message TEXT,
    created_at DATETIME,
    updated_at DATETIME,
    UNIQUE KEY uk_user_business_key (user_id, business_key)
);
```

其中关键是：

```sql
UNIQUE KEY uk_user_business_key (user_id, business_key)
```

含义是：

> 同一个用户的同一个业务请求，只能创建一条任务记录。

如果 3 次相同请求同时进来：

```text
第一次：insert 成功，返回 taskId
第二次：insert 触发 Duplicate Key，查询已有任务并返回原 taskId
第三次：insert 触发 Duplicate Key，查询已有任务并返回原 taskId
```

最终数据库中只会有一条任务。

今天修正了一个理解：

> 数据库唯一索引不是不能扛大数据量，而是在高并发重复请求下，如果所有请求都直接打到数据库，会增加数据库写入和唯一索引冲突压力。

所以数据库唯一索引应该作为强一致兜底，而不是唯一的前置拦截手段。

---

## 4. Redis 和数据库唯一索引的组合

今天我判断应该使用 Redis + 数据库唯一索引组合，这个方向是对的。

现在更准确的理解是：

```text
Redis：前置拦截，降低短时间重复请求对数据库的压力
数据库唯一索引：最终兜底，保证数据层面不会重复创建
```

Redis 可以使用 `SETNX` 做短时间防重复提交：

```java
Boolean first = redisTemplate.opsForValue()
    .setIfAbsent("task:create:" + businessKey, "1", Duration.ofSeconds(10));

if (Boolean.FALSE.equals(first)) {
    // 短时间内已有相同请求，查询已有任务并返回
}
```

但 Redis 不能作为唯一保障，因为 Redis 可能挂掉、过期时间可能设置不合理、也可能出现业务执行和 Redis 标记不一致的问题。

因此更稳妥的设计是：

```text
请求进入
↓
计算 businessKey
↓
Redis SETNX 做短时间防重
↓
数据库 insert 任务
↓
数据库唯一索引兜底
↓
如果唯一索引冲突，就查询已有任务并返回
```

核心结论：

> Redis 是前置防线，数据库唯一索引是最终防线。

## 5. AI 异步任务的正确流程

今天最重要的工程化理解是：

> AI 异步任务不能先调用模型，再保存任务。
> 应该先创建任务记录，再调用模型。

错误流程：

```text
收到请求
↓
调用大模型
↓
保存任务和结果
```

这个流程的问题是，如果模型调用成功，但是数据库保存失败，系统就无法知道这次模型调用已经发生过。下次重试时，可能会再次调用模型，造成重复扣费和资源浪费。

正确流程：

```text
收到请求
↓
计算 businessKey
↓
插入任务记录，状态为 CREATED
↓
返回 taskId
↓
后台 worker 执行任务
↓
状态改为 RUNNING
↓
调用大模型
↓
保存结果
↓
状态改为 SUCCESS
```

推荐状态机：

```text
CREATED → RUNNING → SUCCESS
                  ↘ FAILED
```

每个状态的含义：

| 状态      | 含义             |
| ------- | -------------- |
| CREATED | 任务已创建，还未执行     |
| RUNNING | 任务正在执行         |
| SUCCESS | 任务执行成功，结果已保存   |
| FAILED  | 任务执行失败，已记录错误原因 |

这个流程的核心价值是：

> 先让系统知道任务存在，再去执行高成本的外部模型调用。

## 6. 模型调用失败和保存失败的区别

今天后半部分比较难，但我理解到一个关键点：

> 模型调用失败 和 模型调用成功但保存结果失败，是两个不同问题。

### 6.1 模型调用失败

例如：

* 网络超时
* 模型接口报错
* 参数错误
* 额度不足
* 服务不可用

处理方式：

```text
任务状态 RUNNING
↓
调用模型失败
↓
记录错误原因
↓
状态改为 FAILED
```

例如：

```text
status = FAILED
error_message = "model timeout"
```

这种场景可以后续重试，也可以让用户重新发起任务。

---

### 6.2 模型调用成功，但数据库保存失败

这个场景更复杂。

流程是：

```text
任务状态 RUNNING
↓
模型返回成功
↓
保存 result 到数据库
↓
数据库保存失败
```

这时不能马上重新调用模型，因为模型已经成功执行了，失败点在数据库保存。

正确处理思路是：

```text
模型结果还在内存中
↓
优先重试保存数据库
↓
保存成功后改为 SUCCESS
↓
多次保存失败后记录异常状态
```

核心原则：

> 如果失败的是保存结果，就优先重试保存结果，而不是重新调用模型。

---

## 7. 最坏情况：模型成功后服务宕机

今天最难理解的是这个场景：

```text
模型调用成功
↓
结果还没保存到数据库
↓
服务突然宕机
```

这时会出现：

```text
数据库里任务状态可能还是 RUNNING
但是没有 result
模型结果也丢了
```

这种情况是因为外部模型调用和本地数据库事务无法天然保证一致。

数据库事务只能控制本地数据库操作，不能控制 OpenAI、Claude、千帆等外部模型服务的执行结果。

所以这种场景很难做到 100% 完美，只能通过工程设计降低风险，例如：

* 先创建任务记录，再调用模型。
* 使用任务状态机记录执行进度。
* 模型返回后尽快保存结果。
* 保存失败时优先重试保存。
* 对长时间 RUNNING 的任务做补偿扫描。
* 如果模型服务支持 `Idempotency-Key`，则传入相同 key，避免重复扣费或重复生成。
* 如果模型服务不支持幂等，则只能在失败后标记异常，必要时让用户重试。

今天理解到：

> 这类问题本质上不是简单的幂等问题，而是外部系统调用和本地数据库保存之间的一致性问题。

## 8. 今天形成的标准设计方案

针对 AI 异步任务创建接口：

```http
POST /api/task/create
```

我的设计思路是：

```text
1. 根据 userId、taskType、fileHash、promptHash、model、paramsHash 生成 businessKey。

2. 数据库 ai_task 表对 userId + businessKey 建唯一索引，保证同一个业务请求只能创建一条任务。

3. Redis 使用 SETNX 做短时间防重复提交，拦截 1 秒内多次完全相同的请求，降低数据库压力。

4. Redis 只作为前置防线，数据库唯一索引才是最终兜底。

5. 创建任务时先写数据库，状态为 CREATED，不要先调用模型。

6. 后台 worker 执行任务，把任务状态改为 RUNNING。

7. 调用大模型，成功后保存 result，并把任务状态改为 SUCCESS。

8. 如果模型调用失败，记录 error_message，并把任务状态改为 FAILED。

9. 如果模型调用成功但保存结果失败，优先重试保存结果，而不是重新调用模型。

10. 如果服务在模型成功后、保存结果前宕机，则通过任务状态扫描、失败补偿、模型侧 Idempotency-Key 或人工/自动重试降低损失。
```

## 9. 今天修正的错误理解

### 错误理解一：唯一标识就是 UNIQUE

修正后：

> UNIQUE 是数据库约束，Business Key 才是业务唯一标识。
> 先有合理的 Business Key，UNIQUE 才有意义。

---

### 错误理解二：Redis 可以单独保证幂等

修正后：

> Redis 可以做前置拦截，但不能作为最终一致性保障。
> 数据库唯一索引才是最终兜底。

---

### 错误理解三：数据库唯一索引不能抗大量数据

修正后：

> 数据库唯一索引本身是可靠的幂等手段。
> 真正的问题是高并发重复请求全部打到数据库会增加压力，所以可以用 Redis 提前挡一层。

---

### 错误理解四：模型保存失败后重新调用模型即可

修正后：

> 如果模型已经成功返回，失败点是数据库保存，就应该优先重试保存结果，而不是重新调用模型。
> 否则会重复消耗 token，甚至造成重复扣费。

---

## 10. 今天的最终理解

今天对幂等设计的最终理解是：

> 幂等的核心不是 Redis，也不是 UNIQUE，而是先找到稳定的 Business Key，再用数据库唯一索引做最终兜底，用 Redis 做前置拦截，用状态机控制任务执行过程，用补偿机制处理执行中断和异常情况。

可以进一步总结为：

```text
Business Key：判断是不是同一个业务请求
数据库唯一索引：保证不会重复创建
Redis SETNX：挡住短时间重复提交
状态机：记录任务执行到哪一步
补偿机制：处理中间失败后的收尾问题
```

这次学习让我意识到，真实后端工程不是只写接口和 CRUD，而是要考虑：

* 重复请求
* 并发冲突
* 数据一致性
* 外部系统调用失败
* 本地保存失败
* 成本浪费
* 异常补偿
* 系统最终状态是否可控

这是支付、消息队列、文件转换、AI 异步任务等业务中非常基础但重要的工程能力。
