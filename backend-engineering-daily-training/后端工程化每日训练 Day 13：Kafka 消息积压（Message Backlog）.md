# 后端工程化每日训练 Day 13：Kafka 消息积压（Message Backlog）

## 一、今天学习的知识点

今天学习的是：

**Kafka 消息积压（Message Backlog）**

Kafka 消息积压主要描述：

> 某个 Consumer Group 的消费进度持续落后于 Topic 对应 Partition 的最新写入进度，导致尚未被该消费组处理的消息不断增加。

最常见的情况是：

```text
Producer 生产速度
>
Consumer 实际处理速度
```

例如：

```text
Producer：10000 条/秒
Consumer：3000 条/秒
```

每秒新增积压：

```text
10000 - 3000 = 7000 条
```

一分钟新增积压：

```text
7000 × 60 = 420000 条
```

如果这种速度差长期存在，Kafka 中待消费消息会越来越多，用户任务的等待时间也会持续增加。

但是今天需要修正的第一个认知是：

> 消息积压不一定只是 Producer 突然生产太快，也可能是 Consumer 宕机、消费逻辑变慢、数据库变慢、AI 接口被限流、频繁 Rebalance、热点 Partition 或毒消息持续重试造成的。

所以看到 Lag 上升以后，不能直接判断：

```text
一定是 Producer 生产太快
```

也不能直接判断：

```text
一定需要增加 Worker
```

正确做法应该是先通过指标定位瓶颈，再决定限流、修复、扩容或者优化。

---

## 二、Kafka 为什么会允许消息暂时积压

Kafka 不只是负责传递消息，也承担流量缓冲作用。

例如 AI 异步任务平台：

```text
用户上传 PDF
↓
任务服务创建任务
↓
Producer 发送 Kafka 消息
↓
Kafka 保存待处理任务
↓
PDF Worker 消费任务
↓
执行转换、OCR 或 AI 分析
```

如果高峰期用户突然上传大量文件，Worker 不一定能立刻处理全部任务。

Kafka 可以先保存这些消息，让 Worker 按照自己的处理能力逐步消费。

因此，短时间积压不一定是事故。

例如：

```text
10:00 突然产生 100000 个任务
10:05 Lag 上升
10:20 Worker 已经追平
```

这种情况可能只是正常的流量削峰。

真正需要警惕的是：

```text
Lag 持续增长
```

或者：

```text
Lag 长时间无法下降
```

以及：

```text
消息对应的业务等待时间已经超过用户能够接受的范围
```

所以判断积压是否严重，不能只看当前 Lag 数字，还要看：

```text
Lag 的增长速度
消费速率
生产速率
消息等待时间
业务 SLA
积压恢复时间
```

## 三、消息积压会带来什么业务问题

### 1. 用户任务长时间停留在排队状态

例如 PDF 转换任务：

```text
任务创建成功
↓
状态 = QUEUED
↓
Kafka 中等待
↓
Worker 长时间没有处理
```

用户看到的现象是：

```text
任务提交了几个小时
但是一直没有完成
```

此时系统接口可能没有报错，Kafka 也可能没有丢消息，但是业务体验已经失败。

---

### 2. 业务任务失去时效性

例如广告投放任务要求：

```text
今天 10:00 执行
```

但是因为积压，直到：

```text
今天 17:00 才被消费
```

这时消息可能还存在于 Kafka 中，但任务已经没有执行价值。

这里需要区分两个概念。

```text
业务过期
→ 消息还在 Kafka 中
→ 但是任务已经超过 executeBefore 或业务截止时间
```

```text
Kafka 数据过期
→ 消息超过 Topic 的保留时间或保留空间
→ 数据可能已经被 Kafka 清理
```

因此真实任务消息中可以包含：

```text
createdAt
executeBefore
priority
taskType
```

Consumer 收到消息以后，先判断任务是否还值得执行。

例如：

```java
if (task.getExecuteBefore().isBefore(Instant.now())) {
    taskService.markExpired(task.getId());
    return;
}
```

---

### 3. 为了追赶积压反而压垮下游

发生积压以后，如果直接增加大量 Worker：

```text
大量 Worker 同时消费
↓
大量查询和更新数据库
↓
大量调用 AI 接口
↓
大量下载和上传文件
↓
数据库、第三方服务和网络压力上升
```

可能形成恶性循环：

```text
Worker 增加
↓
下游过载
↓
单个任务耗时变长
↓
实际消费速度下降
↓
Lag 继续上升
```

所以：

> 提高 Kafka 消费速度必须同时考虑下游系统的真实承载能力。

---

### 4. 消息可能在恢复前被清理

如果积压持续时间超过 Topic 的数据保留时间，消费者可能还没有读到消息，消息就已经被 Kafka 清理。

因此严重积压时还要关注：

```text
Topic retention 配置
Broker 磁盘使用率
积压最老消息的等待时间
预计恢复时间
```

不能只关注 Consumer。

## 四、Kafka Lag 的准确含义

Kafka 消息积压最核心的指标是：

**Consumer Lag**

可以简化为：

```text
Partition Lag
=
Log End Offset
-
Consumer Group Committed Offset
```

其中：

```text
Log End Offset
→ 当前 Partition 最新写入位置
```

```text
Committed Offset
→ 某个 Consumer Group 已经提交的消费进度
```

例如：

```text
Log End Offset = 100000
Committed Offset = 80000
```

那么：

```text
Lag ≈ 20000
```

表示这个消费组在该 Partition 上还有大约 20000 条记录没有追上。

这里必须明确三个维度：

```text
Consumer Group
Topic
Partition
```

因为同一个 Topic 可以被多个 Consumer Group 独立消费。

例如：

```text
Topic = pdf-task

Consumer Group A = pdf-convert-group
Consumer Group B = audit-group
```

两个 Group 有各自独立的消费进度。

可能出现：

```text
pdf-convert-group Lag = 500000

audit-group Lag = 100
```

所以不能只说：

```text
这个 Topic 的 Lag 是多少
```

更准确的说法是：

```text
某个 Consumer Group
在某个 Topic
某个 Partition 上的 Lag
```

---

## 五、Lag 是消息条数，不等于时间延迟

我今天需要建立的另一个重要认知是：

> Lag 通常表示尚未追上的消息数量，不直接等于用户已经等待了多长时间。

例如同样是：

```text
Lag = 500000
```

如果净消化速度是：

```text
100000 条/秒
```

可能很快就恢复。

如果净消化速度只有：

```text
100 条/秒
```

则可能需要很长时间。

理论恢复时间可以粗略估算：

```text
恢复时间
≈
当前积压量
÷
（消费速度 - 生产速度）
```

前提是：

```text
消费速度 > 生产速度
```

例如：

```text
当前积压 = 500000
生产速度 = 3000 条/秒
消费速度 = 5000 条/秒
```

净消化速度：

```text
5000 - 3000 = 2000 条/秒
```

理论恢复时间：

```text
500000 ÷ 2000 = 250 秒
```

但是如果：

```text
消费速度 <= 生产速度
```

则：

```text
积压不会被清空
```

甚至还会继续扩大。

所以线上处理积压时，应该同时回答：

```text
当前有多少积压？
Lag 还在增长吗？
每秒净增加多少？
当前净消化速度是多少？
预计多久能够恢复？
```

## 六、Partition 和 Consumer 并行度的关系

Kafka Topic 会被拆分为多个 Partition。

例如：

```text
Topic：pdf-task
Partition 数量：4
```

Consumer Group 中有两个 Consumer：

```text
Consumer A
→ Partition 0、1

Consumer B
→ Partition 2、3
```

如果增加到四个 Consumer：

```text
Consumer A → Partition 0
Consumer B → Partition 1
Consumer C → Partition 2
Consumer D → Partition 3
```

此时可以形成四个 Partition 级别的并行消费单元。

但是如果继续增加到八个 Consumer：

```text
Partition 数量 = 4
Consumer 数量 = 8
```

在同一个传统 Consumer Group 中，同一时刻最多只有四个 Consumer 能够获得 Partition。

剩余 Consumer 没有 Partition 可以消费。

因此：

```text
Consumer Group 的有效并行度
通常不会超过 Partition 数量
```

这就是为什么：

```text
只增加 Worker
但是不考虑 Partition 数量
```

可能完全没有效果。

## 七、Worker、Consumer 和消费线程的关系

在 AI 异步任务平台中：

```text
Worker
→ 承载消费代码并真正执行 PDF、OCR、AI 等任务的服务或进程
```

```text
Consumer
→ Kafka 消费体系中的角色，负责从分配到的 Partition 拉取消息并执行监听逻辑
```

一个 Worker 实例中可以启动多个 Consumer 线程。

Spring Boot 配置例如：

```yaml
spring:
  kafka:
    listener:
      concurrency: 3
```

可以理解为监听容器尝试启动多个并发 Consumer。

但是实际有效并行度仍然受 Partition 数量限制。

例如：

```text
Partition = 2
concurrency = 10
```

实际最多只有两个 Consumer 能获得 Partition。

另外：

```text
Consumer 数量增加
```

不等于：

```text
业务吞吐一定线性增加
```

因为 Consumer 拿到消息以后，还要执行：

```text
数据库查询
文件下载
PDF 转换
AI 接口调用
状态更新
日志记录
```

真正瓶颈可能不在 Kafka，而在这些业务步骤。

## 八、AI 异步任务事故场景

今天的思考题是：

```text
昨天 Kafka Lag = 1000
今天 Kafka Lag = 500000

用户反馈：
PDF 转换任务几个小时没有完成
```

需要判断问题可能发生在：

```text
Producer 生产太快
Consumer 消费太慢
下游数据库变慢
AI 模型接口变慢
```

我一开始的回答是：

```text
可以先看失败日志
也许能够看出一些端倪
```

这个方向不能算错，但是不够完整。

因为日志更适合解释：

```text
具体发生了什么异常
```

但是判断整体生产速度和消费速度是否失衡，首先应该看指标。

正确排查主线是：

```text
先比较生产速率和消费速率
↓
再拆解 Consumer 每个阶段的耗时
↓
再检查数据库和 AI 接口
↓
最后决定限流、修复、扩容还是优化
```

---

## 九、第一步：比较生产速率和消费速率

首先查看：

```text
Producer 每秒写入消息数量
Consumer 每秒处理消息数量
Lag 每分钟增长数量
```

并和历史正常时期进行比较。

### 情况一：生产速率突然上涨

例如：

```text
昨天：
生产速度 = 3000 条/秒
消费速度 = 3500 条/秒

今天：
生产速度 = 10000 条/秒
消费速度 = 3500 条/秒
```

此时 Consumer 能力没有明显下降，主要变化是流量突然上升。

可以判断：

```text
当前积压主要由生产量超过系统设计容量造成
```

此时重点应该考虑：

```text
入口限流
任务准入控制
高低优先级隔离
容量扩展
```

---

### 情况二：生产速率正常，但消费速率下降

例如：

```text
昨天：
生产速度 = 3000 条/秒
消费速度 = 3500 条/秒

今天：
生产速度 = 3000 条/秒
消费速度 = 500 条/秒
```

说明 Producer 没有明显变化，但是 Consumer 实际吞吐下降。

接下来要继续判断：

```text
Consumer 自身异常
还是
Consumer 依赖的下游变慢
```

---

### 情况三：生产和消费都上涨，但消费增长不够

例如：

```text
昨天：
生产 = 3000 条/秒
消费 = 3500 条/秒

今天：
生产 = 10000 条/秒
消费 = 6000 条/秒
```

说明 Consumer 已经提高了处理量，但仍然追不上新流量。

此时不能简单归因于单一故障，可能是：

```text
流量超出系统容量
+
Consumer 扩展能力不足
```

---

## 十、第二步：拆解 Consumer 的任务处理耗时

如果确认消费速率下降，需要继续检查：

```text
一条 PDF 任务到底慢在哪里
```

不能只记录：

```text
任务总耗时 = 120 秒
```

最好拆分成：

```text
数据库读取耗时
文件下载耗时
PDF 转换耗时
OCR 耗时
AI 接口耗时
结果上传耗时
数据库状态更新耗时
任务总耗时
```

例如：

```text
数据库读取：1 秒
文件下载：2 秒
PDF 转换：5 秒
AI 接口：105 秒
状态更新：1 秒
其他：6 秒
```

可以判断：

```text
主要瓶颈在 AI 接口
```

如果是：

```text
数据库读取：70 秒
AI 接口：10 秒
其他：5 秒
```

则应该重点检查数据库。

所以日志不是没有用，而是应该记录：

```text
结构化的阶段耗时
```

而不是只有：

```text
任务失败
```

或者：

```text
处理完成
```

---

## 十一、如何判断数据库是不是瓶颈

如果怀疑数据库，需要查看：

```text
SQL 平均耗时
SQL P95 / P99
慢 SQL 数量
数据库 CPU
磁盘 IO
连接池活跃连接数
获取连接等待时间
锁等待
死锁
事务执行时间
```

例如：

```text
连接池最大连接数 = 50
活跃连接数 = 50
等待连接线程数 = 200
```

说明大量 Worker 正在等待数据库连接。

这时继续增加 Worker：

```text
不会让数据库凭空产生更多处理能力
```

反而可能：

```text
增加等待线程
增加超时
增加锁竞争
增加内存占用
```

另外需要检查：

```text
是不是某条 SQL 突然走了全表扫描
是不是索引失效
是不是批量任务改成了逐条更新
是不是长事务占用连接
是不是多个 Worker 同时更新同一条任务记录
```

---

## 十二、如何判断 AI 模型接口是不是瓶颈

如果 PDF 任务需要调用 AI 模型接口，需要查看：

```text
AI 接口平均响应时间
P95 / P99 延迟
请求超时数量
429 限流数量
5xx 错误数量
当前并发数
QPS 限制
账户额度
重试次数
```

例如：

```text
正常时期平均耗时 = 5 秒
事故期间平均耗时 = 60 秒
同时出现大量 429
```

可以判断：

```text
AI 服务可能被限流
或者当前并发超过接口容量
```

如果 Consumer 遇到 429 后立即高频重试：

```text
请求失败
↓
立即重试
↓
继续触发限流
↓
任务处理时间变长
↓
Consumer 吞吐下降
↓
Lag 增长
```

因此 AI 接口异常时可能需要：

```text
指数退避
并发限制
熔断
备用模型或备用渠道
任务降级
```

而不是简单增加 Worker。

## 十三、为什么要查看每个 Partition 的 Lag

只看 Topic 或 Consumer Group 的总 Lag，可能看不到数据倾斜问题。

### 情况一：所有 Partition 都在上涨

例如：

```text
Partition 0 = 100000
Partition 1 = 120000
Partition 2 = 110000
Partition 3 = 90000
```

通常说明：

```text
整体消费能力不足
```

或者：

```text
所有 Consumer 共同依赖的数据库、AI 接口、文件服务变慢
```

---

### 情况二：只有一个 Partition 特别高

例如：

```text
Partition 0 = 450000
Partition 1 = 1000
Partition 2 = 800
Partition 3 = 1200
```

可能原因包括：

```text
热点 key
分区数据不均衡
负责该 Partition 的 Consumer 异常
某条毒消息持续失败
某类 PDF 特别耗时
```

此时即使增加很多 Worker，其他 Worker 也不能同时消费同一个 Partition 中的消息。

所以需要进一步看：

```text
消息 key 分布
Partition 分配情况
该 Consumer 的日志和资源
是否存在重复失败的 messageId
是否存在超大文件或异常任务
```

---

## 十四、Consumer 自身还可能出现什么问题

除了下游变慢，Consumer 自身也可能导致积压。

例如：

```text
Consumer 实例宕机
线程死锁
Full GC
CPU 打满
内存不足
网络连接异常
消费线程池耗尽
异常消息持续重试
频繁 Rebalance
```

还要关注：

```text
单次业务处理时间是否过长
Consumer 是否长时间没有继续 poll
是否超过 max.poll.interval.ms
```

如果 Consumer 长时间没有继续拉取消息，可能被 Consumer Group 判断为失效，并触发分区重新分配。

频繁 Rebalance 期间：

```text
Partition 需要重新分配
↓
消费暂停或波动
↓
整体吞吐下降
↓
Lag 继续增长
```

所以排查 Consumer 时要查看：

```text
实例数量
实例存活状态
CPU
内存
GC
线程状态
消费异常
Rebalance 次数
poll 间隔
单条任务耗时
```

---

## 十五、临时降低积压的正确处理顺序

我一开始想到的措施包括：

```text
增加 Partition
增加 Worker
增加并发线程
控制生产速度
批量查询
异步调用
优化消费逻辑
```

这些方向中有一部分正确，但是顺序需要修正。

更合理的事故处理顺序是：

```text
限制新增压力
↓
修复直接故障
↓
在容量允许时扩容 Consumer
↓
优化消费逻辑
↓
必要时再评估增加 Partition
```

---

## 十六、第一优先级：限制新增任务进入

如果 Lag 还在快速增长，首先要阻止问题继续扩大。

可以采取：

```text
限制单个用户批量上传数量
限制全局运行中任务数量
限制单位时间提交任务数量
暂停低优先级任务
对非核心客户或功能降级
高低优先级任务分流
```

我一开始说：

```text
超过并发数量就进入等待队列
```

这里需要进一步修正。

Kafka 本身已经是等待队列。

如果只是再创建一个没有容量限制的新队列：

```text
并没有消除积压
只是把积压从 Kafka 转移到另一个地方
```

更准确的设计是：

```text
当前排队任务超过系统安全阈值
↓
限制新任务准入
↓
提示用户稍后提交
或者只允许少量任务进入
```

例如：

```text
单个用户最多允许 20 个 QUEUED / RUNNING 任务
系统总排队任务超过 100000 后暂停大批量上传
```

这属于：

**准入控制和背压。**

---

## 十七、第二优先级：修复直接故障

如果消费速度下降是因为明确故障，应该优先修复故障。

例如：

```text
慢 SQL
→ 优化查询、补充索引或临时关闭非必要查询
```

```text
数据库连接池耗尽
→ 排查连接泄漏、长事务和不合理并发
```

```text
AI 接口大量 429
→ 降低并发、增加退避、切换备用接口
```

```text
Consumer 线程死锁
→ 修复后受控重启
```

```text
毒消息持续失败
→ 隔离到失败队列或人工处理区
```

```text
文件存储异常
→ 恢复存储服务或暂停相关任务
```

发生积压时，重启 Consumer 不是绝对错误。

如果 Consumer 已经：

```text
线程死锁
内存泄漏
连接异常
配置失效
```

受控重启可能是有效的止血措施。

但是不能把重启当作默认答案。

因为重启可能触发：

```text
Rebalance
重复消费
消费暂停
在途任务中断
```

所以正确理解是：

> 先判断 Consumer 是否真的异常，再决定是否受控重启，同时确认 offset 提交、任务幂等和重复执行风险。

---

## 十八、第三优先级：在容量允许时扩容 Consumer

如果确认：

```text
Consumer 数量不足
```

并且：

```text
Partition 数量还有余量
数据库还有容量
AI 接口还有容量
CPU、内存和网络还有容量
```

可以增加 Consumer 或 Worker 实例。

例如：

```text
Partition = 20
Worker = 5
```

可以评估增加到：

```text
Worker = 10
```

但是如果：

```text
Partition = 4
Worker = 4
```

继续增加到：

```text
Worker = 20
```

传统 Consumer Group 中仍然最多只有四个 Consumer 能够拿到 Partition。

另外扩容后要持续观察：

```text
消费速率是否真的上升
数据库延迟是否上升
AI 接口错误率是否上升
Worker CPU 是否合理
Lag 净增长是否转为净下降
```

不能只看：

```text
Worker 数量增加了
```

---

## 十九、第四优先级：优化消费逻辑

可以考虑：

```text
批量查询数据库
批量更新状态
减少重复 IO
减少重复读取配置
缓存低频变化数据
合并日志写入
减少不必要的远程调用
优化 PDF 转换逻辑
按任务大小分流
```

例如原来每处理一条消息都执行：

```text
查询任务
查询用户
查询模型配置
更新任务状态
插入处理记录
```

可以评估：

```text
是否存在重复查询
是否可以批量读取
是否可以减少中间状态更新
是否可以使用缓存
```

但是批量处理也需要控制批次大小。

批次过大可能导致：

```text
单次事务过长
内存占用过高
失败回滚范围过大
消息确认延迟
```

## 二十、为什么不能把异步调用和增加线程池当作通用答案

我一开始提到：

```text
异步调用
增加消费者线程池数量
```

这两个手段不能直接当作安全优化。

例如 Kafka Listener 收到消息后：

```text
把任务提交到业务线程池
↓
Listener 方法立即返回
↓
offset 可能被提交
↓
线程池中的真实任务还没有完成
↓
服务突然宕机
```

可能出现：

```text
Kafka 认为消息已经消费
但是业务任务实际没有完成
```

所以异步消费必须同时设计：

```text
offset 提交时机
任务失败重试
消息幂等
线程池满时的拒绝策略
服务重启后的任务恢复
是否允许乱序
在途任务如何追踪
```

增加线程池也不是越大越好。

如果线程池从：

```text
20 个线程
```

增加到：

```text
500 个线程
```

可能带来：

```text
上下文切换增加
内存占用增加
数据库并发增加
AI 接口并发增加
超时和限流增加
```

所以线程池大小应该根据：

```text
任务是 CPU 密集型还是 IO 密集型
机器 CPU 和内存
下游连接池大小
第三方并发限制
任务平均耗时
```

进行评估。

---

## 二十一、为什么不能第一时间增加 Partition

我一开始回答临时处理措施时，把：

```text
增加 Partition
```

放在了最前面。

这个顺序不合理。

增加 Partition 属于架构和容量变更，不是常规的第一止血手段。

原因包括：

### 1. 当前瓶颈可能根本不在 Partition

如果问题是：

```text
数据库慢 SQL
AI 接口 429
文件存储异常
Consumer 死锁
```

增加 Partition 不会解决问题。

反而可能让更多 Consumer 同时打向已经过载的下游。

---

### 2. 增加 Partition 会带来 Rebalance 和资源开销

Partition 增加后，Consumer Group 需要重新分配 Partition。

Broker 还需要承担更多：

```text
文件句柄
日志段
网络请求
副本同步
元数据管理
```

---

### 3. 可能影响 key 的顺序语义

Kafka 只能保证：

```text
同一个 Partition 内的消息顺序
```

如果 Producer 根据 key 和 Partition 数量计算分区，增加 Partition 后，同一个 key 的后续消息可能被映射到新的 Partition。

这可能影响：

```text
同一个任务
同一个用户
同一个订单
```

相关消息的顺序设计。

---

### 4. Partition 数量通常不能简单恢复原状

增加 Partition 前应该认真评估长期容量，而不是为了短期事故随意修改。

所以更准确的判断是：

> 只有确认现有 Partition 数量已经成为消费并行度瓶颈，并且业务顺序、Broker 容量和下游容量都允许时，才评估增加 Partition。

---

## 二十二、为什么不能无限增加 Worker

我今天对这个问题的回答主要包括两点：

```text
Partition 数量限制有效 Consumer 数量
数据库存在容量上限
```

这两个判断是正确的。

还可以扩展为六类限制。

### 1. Partition 并行度上限

```text
Partition = 4
Worker = 20
```

同一个 Consumer Group 中，最多只有四个 Consumer 能获得 Partition。

其余 Worker 可能处于空闲状态。

---

### 2. 数据库容量上限

更多 Worker 会带来更多：

```text
查询
写入
事务
连接
锁竞争
```

数据库可能出现：

```text
CPU 打满
连接池耗尽
磁盘 IO 升高
慢 SQL 增多
锁等待严重
```

---

### 3. AI 接口和第三方服务限制

AI 模型接口可能存在：

```text
QPS 限制
并发限制
Token 配额
429 限流
账户额度
```

增加 Worker 可能只是制造更多失败请求。

---

### 4. 文件存储和网络限制

PDF 任务可能需要：

```text
从 OBS / OSS 下载文件
上传结果文件
调用 OCR
传输大文件
```

网络带宽和对象存储也可能成为瓶颈。

---

### 5. Worker 机器资源限制

PDF 转换、OCR 和压缩任务可能消耗大量：

```text
CPU
内存
磁盘临时空间
文件句柄
```

实例过多或线程过多可能导致机器本身雪崩。

---

### 6. 成本和系统复杂度

更多 Worker 会增加：

```text
服务器成本
容器资源成本
日志量
监控成本
Rebalance 频率
并发冲突
排障复杂度
```

因此扩容应该基于容量数据，而不是凭感觉无限增加。

---

## 二十三、毒消息如何造成局部积压

有些消息因为数据异常，可能每次消费都失败。

例如：

```text
PDF 文件损坏
文件地址不存在
消息字段缺失
任务状态非法
第三方接口始终拒绝
```

如果 Consumer 对同一条消息无限重试：

```text
消费失败
↓
立即重试
↓
再次失败
↓
继续重试
```

可能导致：

```text
当前 Partition 后续消息长期无法推进
```

此时应该设计：

```text
有限次数重试
指数退避
失败消息隔离
死信 Topic
人工补偿
```

不能让一条永远无法成功的消息持续占用消费能力。

同时需要记录：

```text
messageId
taskId
retryCount
lastError
nextRetryTime
```

## 二十五、Lag 告警应该怎么设计

只设置一个固定阈值：

```text
Lag > 100000 就告警
```

可能不够合理。

因为不同 Topic 的流量差异很大。

更合理的告警可以结合：

```text
绝对 Lag
Lag 增长速度
消息等待时间
预计恢复时间
消费速率是否下降
```

例如：

```text
Lag > 100000
并且连续 10 分钟增长
```

或者：

```text
最老任务等待时间 > 15 分钟
```

或者：

```text
预计恢复时间 > 30 分钟
```

这样比只看一个静态数字更接近真实业务风险。

## 二十六、Spring Kafka 最小并发配置

配置示例：

```yaml
spring:
  kafka:
    consumer:
      group-id: pdf-worker-group
    listener:
      concurrency: 3
```

表示监听容器尝试创建多个并发 Consumer。

但是要注意：

```text
concurrency = 3
```

不代表一定有三倍吞吐。

实际吞吐还受到：

```text
Partition 数量
单条任务耗时
数据库能力
AI 接口能力
机器资源
```

限制。

另外，如果 Listener 内部再把任务提交到业务线程池，需要明确：

```text
offset 何时提交
Listener 返回是否代表业务完成
线程池满了如何处理
任务失败如何恢复
```

## 二十七、常见错误理解

### 错误一：Lag 高就一定是 Producer 有问题

错误。

Lag 高只能说明：

```text
消费进度落后于最新写入进度
```

还需要比较生产和消费速率，并继续排查 Consumer 和下游。

---

### 错误二：看到积压就立即增加 Worker

错误。

应该先判断：

```text
Partition 是否还有并行空间
下游是否有容量
Consumer 是否真的缺实例
```

---

### 错误三：Worker 越多，消费速度一定越快

错误。

超过 Partition 或下游容量以后，Worker 可能空闲，也可能压垮数据库和第三方接口。

---

### 错误四：增加 Partition 是第一止血方案

错误。

增加 Partition 是需要评估顺序语义、Broker 容量和长期架构影响的变更。

---

### 错误五：只看失败日志就能判断根因

不完整。

日志用于解释异常，生产速率、消费速率、Lag、阶段耗时和下游指标才是系统化定位的基础。

---

### 错误六：Lag 就是用户等待时间

错误。

Lag 是数量指标，用户等待时间还和消息产生时间、处理吞吐和业务 SLA 有关。

---

### 错误七：异步提交线程池一定能提高吞吐

错误。

如果 offset 已经提交，但线程池任务尚未完成，服务宕机时可能出现业务任务丢失。

---

### 错误八：重启 Consumer 一定不能做

错误。

受控重启可能解决线程死锁、连接异常等问题，但不能把重启当作默认根治方案。

## 二十八、和之前知识点的联系

Kafka 消息积压可以和之前学习的工程化知识组合起来。

### Day 1：幂等

积压恢复期间可能扩容、重试和重新消费，因此 Consumer 必须保证幂等。

```text
同一个 taskId 重复消费
↓
不能重复扣费
不能重复生成业务结果
不能重复修改最终状态
```

---

### Day 2：消息队列

消息积压体现了 MQ 的流量缓冲价值，也体现了 MQ 并不能无限吸收流量。

---

### Day 3：Consumer ACK

Consumer 什么时候提交消费结果，会影响：

```text
消息是否可能重复消费
业务失败后能否重新处理
异步线程池任务是否可能丢失
```

---

### Day 4：死信队列

毒消息不能无限阻塞正常 Partition，需要通过死信 Topic 或失败任务表隔离。

---

### Day 6：分布式锁

如果多个 Worker 可能重复处理同一个任务，可以结合任务抢占和分布式锁，但不能把锁当成唯一幂等方案。

---

### Day 7：事务一致性与补偿

任务状态更新、消息重试和失败补偿需要保证最终一致性。

---

### Day 8：TraceId 与日志链路追踪

排查单个 PDF 任务时，可以通过：

```text
traceId
taskId
messageId
```

串联：

```text
任务创建
消息发送
Kafka 消费
数据库查询
AI 接口调用
结果更新
```

---

### Day 9：状态机

积压期间任务可能长时间停留在：

```text
QUEUED
```

还需要设计：

```text
QUEUED → RUNNING
QUEUED → EXPIRED
RUNNING → SUCCESS
RUNNING → FAILED
FAILED → RETRYING
```

避免人工随意修改状态。

---

### Day 11：操作日志

管理员如果执行：

```text
暂停消费
恢复消费
手动重试
标记任务过期
切换 AI 模型
```

这些高风险操作应该记录操作日志。

---

### Day 12：Publisher Confirm

Publisher Confirm 解决的是 RabbitMQ Producer 到 Broker 的可靠性问题。

而今天的 Kafka 消息积压关注的是：

```text
消息已经进入 MQ 以后
消费进度为什么落后
以及如何安全恢复
```

两者都属于消息链路可靠性，但责任边界不同。

---

## 二十九、思考题的完整回答

可以这样回答今天的事故题：

> 首先，我不会一看到 Kafka Lag 从 1000 增加到 500000 就直接扩容 Worker，而是先比较生产速率、消费速率以及 Lag 的增长速度。
>
> 如果生产速率相比历史数据突然上涨，而消费速率基本没有变化，说明可能是短时间内任务量增加，Producer 生产速度超过了当前系统的消费能力。如果生产速率没有明显变化，但是消费速率突然下降，则更可能是 Consumer 本身或者下游依赖出现了问题。
>
> 接下来我会查看 Consumer 的任务处理耗时，并把一次 PDF 任务拆分成数据库查询、文件下载、PDF 转换、AI 接口调用和状态更新等阶段。如果数据库查询耗时明显上升，我会继续检查慢 SQL、连接池、锁等待、数据库 CPU 和磁盘 IO；如果 AI 接口耗时增加，并且出现大量超时、429 或 5xx 错误，就说明 AI 模型接口可能变慢或者被限流。
>
> 我还会查看每个 Partition 的 Lag。如果所有 Partition 的 Lag 都在增长，通常说明整体消费能力不足或者公共下游出现问题；如果只有某个 Partition 的 Lag 特别高，则需要排查热点 key、数据倾斜、毒消息或者负责该 Partition 的 Consumer 是否异常。
>
> 如果需要临时降低积压，第一步是限制新任务进入，例如限制用户批量上传数量、限制运行中任务数量，或者暂停低优先级任务，防止积压继续扩大。第二步是修复直接故障，比如处理慢 SQL、恢复 AI 接口、隔离持续失败的消息。第三步是在 Partition 数量和下游容量允许的情况下增加 Consumer 或 Worker 实例。
>
> 同时，我可以通过批量查询、批量更新、减少重复 IO 和优化 PDF 处理逻辑来提高单个 Consumer 的处理效率。增加线程池或者改成异步处理时，需要注意 offset 提交、失败重试、线程池拒绝策略和消息幂等，不能让 Kafka 认为消息已经完成，但实际任务还没有处理完。
>
> 不能无限增加 Worker，首先是因为 Kafka 的有效消费并行度受到 Partition 数量限制，Worker 数量超过 Partition 数量后，部分 Worker 无法分配到 Partition。其次，数据库、AI 接口、文件存储、CPU、内存和网络都有容量上限。大量增加 Worker 可能压垮数据库或者触发第三方接口限流，导致单个任务处理时间变长，最终消费速度不升反降。因此，扩容必须结合 Partition 数量、下游容量和系统监控数据进行。

## 三十、面试表达

可以这样表达：

> 在异步任务系统中，我会重点关注 Consumer Group Lag、生产速率、消费速率和任务等待时间。出现积压后，我不会直接扩容 Worker，而是先比较生产和消费速率，判断是流量突然上涨，还是 Consumer 吞吐下降。
>
> 如果消费速率下降，我会继续拆分单条任务的数据库查询、文件处理、AI 接口调用和状态更新耗时，并结合数据库连接池、慢 SQL、AI 接口 P95、429、5xx、Consumer 资源和 Rebalance 情况定位瓶颈。我还会查看每个 Partition 的 Lag，判断是否存在热点 key、数据倾斜或毒消息。
>
> 临时处理时，我会先限制新增任务和低优先级流量，再修复直接故障，然后在 Partition 和下游容量允许的情况下扩容 Consumer。不能无限增加 Worker，因为有效消费并行度受 Partition 数量限制，同时数据库、第三方接口、存储、CPU 和网络都有容量上限。为了追赶积压而无脑扩容，可能压垮下游，导致消费速度反而下降。

## 三十一、今天最大的认知修正

今天主要有七个认知修正。

### 第一

原来的理解：

```text
消息积压就是 Producer 生产太快
```

修正后：

```text
消息积压只说明消费进度落后
原因可能在 Producer、Consumer、数据库、AI 接口、Partition 分布或重试机制
```

---

### 第二

原来的理解：

```text
看失败日志就可以判断是谁的问题
```

修正后：

```text
先比较生产速率和消费速率
日志和阶段耗时用于继续定位具体瓶颈
```

---

### 第三

原来的处理顺序：

```text
先增加 Partition
再增加 Worker
```

修正后：

```text
先限制新增压力
再修复直接故障
再安全扩容 Consumer
最后才评估 Partition 是否真的不足
```

---

### 第四

原来的理解：

```text
增加 Worker 就可以提高处理速度
```

修正后：

```text
Worker 的有效并行度受 Partition 数量限制
实际吞吐还受数据库、AI 接口、存储、CPU、内存和网络限制
```

---

### 第五

原来的理解：

```text
超过并发任务就再放进等待队列
```

修正后：

```text
Kafka 本身已经是等待队列
真正需要的是任务准入控制、背压和容量限制
```

---

### 第六

原来的理解：

```text
异步调用和增大线程池属于直接优化
```

修正后：

```text
异步化必须同时处理 offset 提交、幂等、失败恢复、线程池拒绝和任务丢失风险
```

---

### 第七

原来的理解：

```text
消息过期就是任务执行晚了
```

修正后：

```text
需要区分业务任务失效和 Kafka 数据被 retention 清理
```

## 三十二、最终理解

我目前对 Kafka 消息积压的理解是：

> Kafka 消息积压本质上是某个 Consumer Group 的消费进度持续落后于 Topic 对应 Partition 的最新写入进度。Lag 上升并不能直接证明 Producer、Consumer、数据库或 AI 接口中的哪一方出现了问题，需要先比较生产速率、消费速率和 Lag 增长速度，再拆分 Consumer 每个业务阶段的耗时，并结合每个 Partition 的 Lag、数据库连接和慢 SQL、AI 接口延迟和限流、Consumer 资源以及 Rebalance 情况定位根因。
>
> 临时处理积压时，正确顺序不是直接增加 Partition 或无限扩容 Worker，而是先限制新增压力，修复直接故障，再在 Partition 数量和下游容量允许的情况下扩容 Consumer，并通过批量处理、减少重复 IO、隔离毒消息和优化业务逻辑提升消费效率。Worker 数量超过 Partition 数量以后，部分 Worker 无法获得 Partition；即使 Partition 足够，数据库、AI 接口、文件存储、CPU、内存和网络也会形成新的容量上限。
>
> 因此，Kafka 消息积压处理不是单纯的 MQ 配置问题，而是生产流量、消费并行度、业务处理耗时、下游容量、任务时效、监控告警和事故恢复共同组成的系统容量与稳定性问题。
